A Distributed DataBase System (DDBS) is a collection of multiple logically related databases distributed over a computer network.


DDBS is a collection of database systems together with software providing a required set of operations on data and management features.


We distinguish two type of distributed database systems: homogeneous DDBS and heterogeneous DDBS.


Homogeneous DDBS is a collection of identical database systems where database management software is coming from the same vendor and distributed over a computer network, e.g. a collection of Oracle database systems.


Heterogeneous DDBS is a collection of different database systems where a database management software is coming from different vendors and it is distributed over a computer network, e.g. a collection of Oracle + MySQL + DB/2 + MongoDB + XML native database systems + Excel spreadsheets + ... , some other systems.

The current slide shows a sample organization of heterogeneous distributed database system. A heterogeneous database systems integrate a relational database system, NoSQL database system, XML native database systems and ordinary file system. NoSQL database system is a database system that provides a logical view of data different from relational view. XML native database system is a system that provides XML as logical view of data.


All components of distributed database system are geographically distributed over different locations all over the world. Each system is connected to its own local area network. It is possible that there are more heterogeneous database systems connected to a local area network. Local area networks are connected to wide area network that allows for communication of geographically distributed local database systems.


Heterogeneity of different database systems has a number of dimensions. Database systems participating in a distributed database system can provide different logical views of data stored in each system. It is so called data model heterogeneity.


Database systems participating in a distributed database system can be implemented with software provided by different vendors. It is so call interpretation heterogeneity. Data kept in database systems participating in a distributed database system may have different semantics, for examplea distance can be measured in kilometres in one database and and in miles in another database.

The database schemas of centralized database system show data at several level of abstraction. At the lowest level of abstraction and the closest to pure hardware level, a physical schema shows a database as a collection of files located at a number of persistent storage devices. The internal structures of such files determine persistent storage allocation strategy, a way how rows are stored in data blocks (row-by-row method versus column-by column method), persistent storage structures of indices or clusters, etc.


At logical level a database systems provide an abstract view of data in the terms of particular data structures like tables, data graphs, trees, etc.


A conceptual schema level provides the topmost view of data in terms of real world objects described by the attributes, associations between the classes of objects and generalizations over the classes of objects. A conceptual schema, usually a diagram, provides information about the contents of a database in the terms of a fragment of the reality described by a database.


Entire conceptual schema is usually to extensive for the individual users. For example a conceptual schema of an entire industrial organization is too large for users from finance department, hr department, manufacturing department and so on. It is why entire conceptual schema of a centralized database is partitioned into the external schemas and such external schemas are presented to the individual groups of users.


In a centralized database schema model the users are provided their personal external schemas (views) of data


The external schemas are integrated into a single conceptual schema later on transformed into a logical schema and implemented as a physical schema


Centralized database schema model

  • In a centralized database schema model the users are provided their personal external schemas (views) of data
  • The external schemas are integrated into a single conceptual schema later on transformed into a logical schema and implemented as a physical schema

We start a review of different architectures of centralized and distributed database systems from mainframe application architecture typical for a period of time between 1960s to early 1980s.


In such approach database systems architecture was strongly determined by the available hardware. In a typical hardware configuration so called "dumb terminals" are directly connected to a mainframe computer. A single centralized database server is located at a mainframe computer. The users in front of "dumb terminals are directly connected to a database server.


All presentation logic of database applications (command line and sometimes graphical interface) and business logic (implementation of database application) are located on a mainframe computer. All data is located on the persistent storage devices directly connected to a mainframe computer.


Mainframe application architecture

  • In mainframe application architecture "dumb" terminals are connected to a single database server

In a client-server architecture "dumb terminals" are replaced with the limited computational power personal computers (starting in mid 1980s). Then entire presentation logic (user interface) moved to a personal computer playing mainly a role of an intelligent terminal. Some simple components of business logic have been also moved to personal computer.


Entire functionality of a database system is implemented as a database server. A database server is a dedicated piece of software responsible for storing and maintenance of data. The remaining components of the business logic are stored in a data dictionary of a database server. Distribution of business logic between clients running on personal computers and database server running on a powerful hardware system allowed for a better balance of overall workload and reduction of the amounts of data transferred from a database server to a client (due to some business logic earlier stored and now running on a database server).


Client-server architecture

  • In client-server architecture personal systems communicate with a single database server

At this point in time, together with the progresses of networking technologies, it was quite easy to upgrade a typical client-server architecture to distributed client-server architecture.


In distributed client server architecture we add more database servers to the system and we allow the clients to connect to any number of database servers at a time. Like before, presentation logic is entirely located at the clients and business logic is distributed between the clients and database servers.


An important improvement when compared with pure client-server architecture is the ability of database servers to communicated with each other and to transfer data from one database to another. For example if two or more database servers support a relational database model then it is possible to join two or more relational tables located at different database servers.


Distributed client-server architecture

  • In distributed client-server architecture personal systems communicate with many database servers

An important progress step in the development of distributed database systems is Web based architecture.


Like in the previous two cases of client-server and distributed client-server architectures all presentation logic is removed from database servers.


In Web based database architecture presentation logic is distributed between the client systems and Web servers also called as application servers.


Business logic is distributed between application servers and database servers. Application servers are build on top of ordinary Web servers. The clients access application servers and pass their request to access/process data to application servers. Then application servers "cooperate" with database servers on passing the clients' requests and on transferring data back to the clients.


Like in the client-server architecture database servers have the abilities to communicate and to pass data between each other.


A layer of application servers between clients and database servers reduces processing load on database servers and provides more persistent and transient storage to be used for data caching.


Web based architecture

  • In Web based architecture personal systems communicate with the Web servers that communicate with a single database server

Distributed database schema architecture integrates the local conceptual schemas of each database system participating in distributed database system into a single global conceptual schema. However, the local physical, logical, and conceptual schmas remain without changes.


A new global conceptual schema hides the distribution details of individual database systems participating in a distributed database system. The users are unable to distinguish between individual database systems while looking at a global conceptual schema.


Individual users obtain external views of a global conceptual schema depending on their interests in the particular subsets of global conceptual schema.


Distributed database schema architecture

  • In distributed database schema architecture a global conceptual schema hides distribution from the users
  • The users can see a distributed database system as a single monolithic database system

A very important concept in organization of distributed database system is "transparency".


Transparency means hiding information from end users. Ther are few very different categories of transparency such as: data organization transparency, replication transparency, fragmentation transparency, design transparency and execution transparency.


Data organization (distribution or network) transparency means hiding network related information and data placement information; it is either location or naming transparency. In location transparency the users do not know the location details distributed database systems.


A global naming procedure is applied to the data objects located in different database systems. It is an example of naming transparency.


Location transparency hides information about the locations of data fragments in the database systems participating in a distributed system. Location transparency allows the operations to be independent on the locations of data objects.


Replication transparency hides from the users information about data replicated in many database systems of a distributed system. Replication transparency means that users are unaware of the existence of multiple copies of the same data objects


Fragmentation transparency hides from the users information about fragmentation data objects, for example vertical projections of one relational table located at different sites of distributed system. Fragmentation transparency means that users are unaware of data fragmentation over many sites. It is implement as combination of vertical and horizontal fragmentation.


Vertical fragmentation picks selected columns from a relational table together with a primary key column(s). Vertical fragments are located at the database systems depending on how frequently are used. Vertical fragments can be replicated in many database systems.


Horizontal fragmentation picks subsets of the rows from a relational table. The horizontal fragments are located at the database systems depending on how frequently are used. Horizontal fragments can be replicated in many database systems.


Transparency

  • Transparency means hiding information from end users
  • Data organization (distribution or network) transparency means hiding network related information and data placement information; it is either location or naming transparency
  • Naming transparency allows for global naming of data objects
  • Location transparency allows the operations to be independent on the locations of data objects
  • Replication transparency means that users are unaware of the existence of multiple copies of the same data objects
  • Fragmentation transparency means that users are unaware of data fragmentation over many sites; it includes vertical and horizontal fragmentation

Another very important concept in organization of distributed database system is a level of "autonomy" granted to each database system participating a distributed system.


Autonomy determines a level of independence of individual nodes in distributed database system. A database system being a member of distributed database system may have very low degree of autonomy. Then, all its actions depend on a coordinator node of a distributed database system. High degree autonomy is required for flexibility and customized maintenance of distributed database system.


The dimensions of autonomy include: design, communication and execution autonomy.


Design autonomy means a level of independence of data model usage and transaction management technique between the nodes.


Communication autonomy means a level of independence to which a node can share information with other nodes


Execution autonomy means a level of independence to which users act as they please.


Autonomy

  • Autonomy determines a level of independence of individual nodes in distributed database system
  • High degree autonomy is required for flexibility and customized maintenance of distributed database system
  • Design autonomy means a level of independence of data model usage and transaction management technique between the nodes
  • Communication autonomy means a level of independence to which a node can share information with other nodes
  • Execution autonomy means a level of independence to which users act as they please

Distributed database systems have their advantages and disadvantages.


The advantages include:

Higher level of reliability and availability

Improved ease and flexibility of application development

Improved performance

Easier expansion


Higher level of reliability and availability is possible when some data is replicated in different sites of distributed database system. Then, if one of the sites is unavailable due to heavy load or network problem it is always possible to get data from another site. If data is replicated in another site then such replication can be considered as some sort of a distributed backup that can be used in emergency situation.


Improved easy and flexibility of application development is possible when data is replicated in several site and some of these site are considered as productions sites while the other are development sites. Then there is no need suspend a production system because of application testing.


Distribution of data over many sites allows for parallelization of data processing in a number of sites. Also if data is not available at one site then it can be accessed in another site. Parallelization of data processing allows for improved performance.


A distributed database system always implement simple the expansion mechanisms through addition of new database sites.


Advantages and disadvantages of DDBS

  • Higher level of reliability and availability
  • Improved ease and flexibility of application development
  • Improved performance
  • Easier expansion

Distributed database systems have their advantages and disadvantages.


The disadvantages include:

Keeping track of data distribution

Distributed query processing

Distributed transaction management

Replicated data management

Distributed database recovery

Security

Distributed directory (catalog) management


Whenever data is distributed over a number of sites in a distributed database system it is necessary to keep information about the locations and replications of data items. Such information must be frequently updated when data is transferred between the sites and/or new replicates are created and/or old replicase a removed. Maintenance of information on data distribution takes additional time and imposed an additional workload on the database systems participating in a distributed system.


Optimization of query processing in a distributed system is a complex problem. Distribution and replication contributes to more query processing plans generate and evaluated by a query optimizer. It slows down optimization of query processing.


Distributed transaction processing is a difficult problem because of synchronization of many different transaction processing systems. For example, synchornization of transaction scheduling based on 2phase locking protocol in one database site and timestamp ordering protocol at another database site.


Another difficult problem is management of replicated data. Enforcement of strict consistency constraints on replicated data significantly slows dow distributed transaction processing.


The same performance problems occur in distributed database recovery. If one of subtransaction of distributed database transactions fails and it must be rolled back then all other subtransactions must be rolled back as well. A failure of a database site participating in a distributed system may require synchronization with the contents of the other sites during a recovery process.


Due to data replication more data is exposed to security risks. Data security protection can be implemented many different ways database sites participating in a distributed system. It makes coordination of data security policies and processes much harder than in a single centralized system.


The importance of distributed directory was already mentioned in a context of keeping information about location and replication of data items.


Advantages and disadvantages of DDBS

  • Keeping track of data distribution
  • Distributed query processing
  • Distributed transaction management
  • Replicated data management
  • Distributed database recovery
  • Security
  • Distributed directory (catalog) management

A database link is a connection between two physical database servers that allow a user to access them as one logical database. A data dictionary view contains the names of database links. It is possible to list it in the following way:


SELECT * FROM USER_DB_LINKS;


Assume, that we are connected as a user scott to a database server DATA-PC01. Then, it is possible to create a database link to a user jrg located at a database server DATA-PC07 in the following way


CREATE DATABASE LINK "DB.DATA-PC07"

CONNECT TO jrg

IDENTIFIED BY rat

USING 'data-pc07.adeis.uow.edu.au:1521/db';


A name of database link ( "DB.DATA-PC07" ) to a database must be the same as a global database name of the database linked to

A connection string ( 'data-pc07.adeis.uow.edu.au:1521/db' ) determines a physical location of the database linked to. It consists of host name in a network (data-pc07.adeis.uow.edu.au) where a database server is located, port number (1521) a listener is listening to, and system identifier (db) of a database system we would like to create a link to.


When a database link is created it is possible to access a relational table located in a database system linked to in the following way:


SELECT *
