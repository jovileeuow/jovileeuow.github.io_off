What is JSON Schema ?


In the past we learnt and we used few other sort of schemas, like for example conceptual schema, relational schema and XML schema.

A schema represents the structures of certain domain expressed in a formalized language. A conceptual schema represents a conceptual structure of a database domain expressed in a graphical notation of simplified UML classes or a graphical notation of Entity-Relationship data model or a notation of Object-Role-Model and many others. A relational schema represents a structure of relational table expressed in a language of named sets of attributes, primary keys, candidate keys and foreign keys. XML schema represents a structure of XML documents expressed in XML notation.


So, again, what is JSON schema.?

JSON schema represents a structure of JSON/BSON document expressed as a JSON document. Hence, like with XML schema where a schema is a well formed XML document , JSON schema is also well formed JSON document.


Why do we need JSON Schema ? As we mentioned before the ideas of unstructured or semistructured data model allow fro lack of rigid schema precisely determining the structures of data objects. In theory such idea seems to be very attractive. Unfortunately in practice when a data object has a structure very different from the majority of data object it is impossible distinguish between such exotic structure or a simple mistake. It means that we have to provide information about a number of possible correct structures instead of saying that any structure will do. It is why we need JSON Schema for validation of JSON objects and also XML schema for validation of XML documents. From such perspective JSON Schema is a vocabulary for validation of JSON objects and MongoDB uses an slightly modified notation of JSON Schema to validate BCON documents in MongoDB collections of documents. A collection of BSON documents can have JSON Schema assigned to more precisely determines the structures of BSON documents stored in the collection.


In a general case, JSON Schema describes the structures and types of existing JSON objects. JSON Schema allows for automated validation of JSON object.


Like XML Schema being itself a well formed XML document, JSON Schema itself is a well formed JSON object.


The latest version of JSON Schema is a version 2020-12


JSON schema expresses the constraints imposed on JSON object. If JSON Schema is an empty object then it positively validates any possible JSON object.


The following JSON object:

{"city":"Sydney", "street":"Victoria", "building number":25, "building name":"James Bond Building"}

validates well against the following JSON Schema:

{ "type":"object",

"properties": {

"city": { "type":"string" },

"street": { "type":"string" },

"building": { "type" :"number",

"minimum": 1}

}}

Note that a JSON Schema above does not impose an constraints for a key:vale pairs "building name":"James Bond Building" or any other

key:vale pair that can be added to the JSON object. It means that the given document can be extended by any additional key:vale pairs

without violation of the constraints imposed by the given JSON Schema.

The simplest possible JSON schema is an empty JSON object.

{ }

Such schema does not enforce any constraints and it validates any well formed JSON object


The extensions of the schema given above impose the restrictions on the validated objects. For example, the following JSON schema

{ "type":"object",

"properties": { "Hello world message": { "type":"string" } }

}

validates well the following JSON objects

{ "Hello world message":"Hello world !" }

{ "Hello world message":"Hello world !", "Greeting":"How are you ?" }

{}


An object that has a key:vale pair "Hello world message":"Hello world !" validates well because a value associated with a key is of type string.


An object { "Hello world message":"Hello world !", "Greeting":"How are you ?" } validates well because like before a value associated with a key is a string and a JSON Schema does not impose any constraints of a key "Greetings".


An empty objet {} validates well because a default validation of a missing key against a key in JSN scheme is always true. It is like a default NULL constraints imposed on a column in CREATE TABLE statements. It is possible to enforce mandatory occurrence of a key:vale pair in a JSON document. Such possibility will be discussed in the future.


Validation with JSON schema applies only to key:value pairs in an object that are listed in JSON schema. If a key:value pair is not listed in JSON schemas then validation does not apply to such key:value pair.


If a key: value pair is listed in JSON schema and it is not included in an object then according to a default interpretation such key:value pair is optional in the object. A default interpretation cam be changed with application of a key "requiredProperties" in JSON schema. This topic will be discussed in the near future.

To determine a type of value associated with a key JSON schema uses a key:vale pair "type":basic-type-name.

JSON schema defines the following basic types: "string", "Numeric types", "object", "array", "boolean", and "null" "Numeric types" will be discussed in the future. "null" means that only no value (null) can be assigned with a key.


If "type" keyword is associated with a name of a basic type then a respective value must be of the same basic type, for example

{ "type":"string"}

determines a type of a respective value as a string.


It is possible to define the variants of types. A basic type associated with a keyword "type" can be either one of the types listed above or it can be an array of the basic types. If "type" keyword is associated with an array of basic types then each element of the array must be unique, for example

{ "type": ["string","number"] }

Such key:vale pair determines a type of a respective value as either string or number

A pattern restriction can be used to used to match a string with a given regular expression.

For example

"number":{"type":"string",

"pattern":"[1-9][0-9]" }

validates well any string of characters representing a number in a range from "10" to "99" with a key "number".


The first [1-9] range subexpression in a regular expression determines the first character to be in a range from "1" to "9". The second [0-9] subexpression determines the second character to be in a range from "0" to "9".


A format restriction is used to match a string with predefined format

For example

"birthday":{"type":"string",

"format":"date" }

validates well any string of characters representing a date in a format "YYYY-MM-DD" for example "2020-07-21" with key "birthday".


The following built-in formats are allowed:

Dates and times

Email addresses

Hostnames

IP Addresses

Resource identifiers

URI template

JSON Pointer

Regular Expressions

JSON Schema has two numeric types: "integer" and "number". A type integer is used for validation of integer numbers

For example

"int":{ "type":"integer" }

validates well any integer number like -5, 0, 7, etc with key "int".


A type "number" is used for validation of any numeric value, either integer or floating point numbers

For example

"num":{ "type":"number" }

validates well any integer or floating point number like like -35.7, -7, 0.0, 23, 23.4, etc with a key "num".


Similarly to "string" type the numeric types can have the restrictions to impose the additional constraints on the values. The restrictions of numeric types are explained in the next slide.

A key "dependencies" determine the existence dependencies of one property on another. For example if a key "a" is in an object then both keys "b" and "c" must be present in the same object. However, if a key "a" is not present in. an object then the keys "b" and "c" can be present or not.


A value associated with a key "dependencies" is an object.


Each entry in the object maps from a name of a property into an array of properties that are required whenever the property is present

For example JSON schema

{ "type":"object",

"properties": { "city": { "type":"string" },

"street": { "type":"string" },

"building": { "type":"number",

"minimum":1 }

"dependencies": { "street": ["building"]} }


validates well the objects

{ "city":"Dapto","street":"Station","building":7}

{ "street":"Station","building":7}

{ "city":"Dapto","building":7}

It is because an object has no key "street".


It does not validate well an object

{ "city":"Dapto","street":"Station"}

It is because a key "street" is present and a key "building" required together with a key "street" is absent.
